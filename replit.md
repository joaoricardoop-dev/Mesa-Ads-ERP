# Mesa Ads ERP

## Overview
Mesa Ads ERP is a financial simulation and management SaaS designed for a Brazilian offline media company specializing in advertising on coasters. Its primary goal is to streamline operations, manage advertising campaigns from lead to invoicing, track finances, and facilitate onboarding for both advertisers and restaurant partners. Key capabilities include a comprehensive campaign workflow, quotation management, lead tracking with auto-classification, a detailed financial dashboard, a multi-dimensional restaurant rating system, and a flexible multi-product pricing engine. The project aims to enhance efficiency, provide robust financial insights, and support business growth in the niche advertising market.

### Fonte única de verdade (OBRIGATÓRIO)
Uma informação = uma origem. O agente repetidamente cria múltiplas fontes para o mesmo dado (constantes, configs, cálculos, mapeamentos, derivações), gerando divergências entre telas (ex.: a mesma data calculada com âncoras diferentes em três pontos). Para evitar isso, toda implementação DEVE seguir estas regras checáveis:

1. **Pesquisar antes de criar.** Antes de adicionar qualquer constante, config, cálculo, mapeamento ou derivação, pesquise (`rg`/grep) se aquele dado já existe no código. Se existir, reuse — não recrie.
2. **Um dado, uma origem.** Todo valor derivado sai de uma única função/fonte canônica, lida por TODAS as telas (interna, pública, PDF). Telas nunca recalculam por conta própria.
3. **Proibido recalcular em mais de um lugar.** É proibido reimplementar o mesmo cálculo/derivação com lógica ou âncora própria em pontos diferentes. Extraia para `shared/` (ou um helper canônico) e chame em todos os pontos.
4. **Na dúvida, pergunte.** Se não tiver certeza se um valor novo é o mesmo dado de um já existente, pergunte ao usuário antes de duplicar.
5. **Liste as fontes ao concluir.** Ao finalizar uma feature, liste no resumo as fontes de dado usadas, confirmando explicitamente que não há duplicação de origem.

## System Architecture

### UI/UX Decisions
The application utilizes a sidebar-based layout built with `shadcn/ui`, offering a collapsible and resizable navigation experience. It supports both light and dark modes, **defaulting to light** (Mesa.ads paper identity), and employs the Mesa.ads brand palette: neon green (`#00c238` light / `#00e640` dark) as primary signal, magenta (`#e51e75` / `#ff2e8a`) as destructive/accent, plus amber/ice/orange for charts. Surfaces use warm paper neutrals in light mode and deep ink (`#0a0a0c`) in dark, with hairline borders (`rgba(0,0,0,0.08)` / `rgba(255,255,255,0.08)`). Typography combines DM Sans (body), Bricolage Grotesque (`font-display` utility for titles), Instrument Serif italic (`font-serif-display` for expressive accents) and JetBrains Mono (`label-mono` utility for uppercase labels and badges). Reusable components like `PageContainer` and `Section` maintain design consistency. The design is mobile-first.

### Technical Implementations
- **Frontend**: React 19, Vite, Tailwind CSS 4, shadcn/ui, TanStack Query, tRPC, Recharts, wouter.
- **Backend**: Node.js, Express, tRPC, Drizzle ORM.
- **Authentication**: Clerk for identity management, roles (Admin, Comercial, Operações, Financeiro, Anunciante, Restaurante) are managed via Clerk's `publicMetadata.role` and synced to a local `users` table. Impersonation functionality is available for internal users.
- **Database**: PostgreSQL, powered by Neon Database (serverless), with Drizzle ORM for type-safe interactions.
- **Module-based Structure**: Organized into distinct modules (Comercial, Financeiro, Campanhas, Restaurantes, Biblioteca), each with dedicated tRPC routers and UI sections.
- **Custom Database Migrations**: A custom migration runner in `server/migrations.ts` handles all database schema changes using idempotent SQL. Drizzle CLI generated files are for reference only.
- **Campaign Workflow**: A 7-step automated pipeline (`briefing` → `design` → `aprovacao` → `producao` → `distribuicao` → `veiculacao` → `inativa`) with SLA tracking and automated production/distribution OS generation. Supports multiple freight trackings per service order.
- **Quotation Workflow**: Manages quotations through statuses (`rascunho` → `enviada` → `ativa` → `os_gerada` → `win` / `perdida` / `expirada`), supporting auto-naming, batch selection, and conversion to campaigns upon digital signing.
- **Financial Dashboard**: Comprehensive dashboard (`/financeiro/dashboard`) with executive KPIs, DRE, detailed revenue analysis, efficiency metrics (quotation funnel), and client performance. Includes server-side alerts based on financial metrics.
- **DRE Dual (Finrefac #7)**: Two regimes — Competência (revenue=issueDate, costs=competenceMonth) and Caixa (revenue=paymentDate, costs=paymentDate). User preference persisted in `users.preferences` jsonb. Dashboard exposes new cards: DSO with 90d trend, inadimplência por aging (0-30/30-60/60-90/90+), funil cotação→fatura→recebido. Relatório financeiro exporta CSV e PDF (jspdf + autotable) com regime selecionado.
- **Restaurant Onboarding**: Supports both self-service public registration and invite-based onboarding, including legal term acceptance with version tracking and profile photo uploads to Replit Object Storage.
- **Mapa de inventário (CatalogMap) e Google Maps**: O builder `/montar-campanha` (e o Orçamento interno) exibem os locais com coordenadas num mapa Google (`InventoryCatalog → CatalogMap`), plotando um pin por local via `google.maps.Marker` e abrindo um `InfoWindow` (DOM via `textContent`, nunca HTML interpolado) com preço/período (fonte única `locationMetrics`) e botão Adicionar/Remover. O loader único é `client/src/lib/googleMaps.ts` (`loadGoogleMaps()`) — use SEMPRE ele para carregar o SDK; ele curto-circuita se `window.google.maps` já existir. **Comportamento da chave (`VITE_GOOGLE_MAPS_API_KEY`, secret):** (a) **ausente** → `loadGoogleMaps()` rejeita com `GoogleMapsKeyMissingError` ANTES de injetar o script (nunca injeta `key=undefined`); `MapView` captura e renderiza um fallback `data-testid="map-load-error"` ("Mapa indisponível…"); o restante da tela (lista/cards) segue funcionando. (b) **inválida/sem billing** → o script carrega (onload dispara), mas o Google sobrepõe seu próprio overlay (`.gm-err-container`, "This page can't load Google Maps correctly") e chama `window.gm_authFailure`, que registramos para logar erro claro no console e setar `window._gmapsAuthFailed`. **Fonte única do loader:** todas as telas com mapa (incl. `client/src/pages/RestaurantsMap.tsx`) usam `loadGoogleMaps()` — não há mais loader paralelo; o comportamento de chave (ausente → fallback; inválida → `gm_authFailure`) é idêntico em todas elas. O `RestaurantsMap` captura a rejeição de `loadGoogleMaps()` e mostra seu próprio fallback (botão "Tentar novamente" reseta `window._gmapsLoading`/`window._gmapsAuthFailed` e re-chama `initMap`).
- **Smoke do mapa contra o SDK real (Task #338)**: `e2e/builder-locais.spec.ts` cobre o CatalogMap com um `window.google.maps` FAKE (markers/InfoWindow viram DOM), provando a fiação data→markers→popup→add-to-plan de forma determinística. O SDK REAL é coberto por `e2e/builder-locais-real-maps.spec.ts`, GATED por `RUN_REAL_MAPS=1` (opt-in; custo/billing + flakiness de rede) e dependente de `VITE_GOOGLE_MAPS_API_KEY` no dev server. Sem a flag, ele é pulado (skip) e a suíte padrão usa só o mock. Rodar: `RUN_REAL_MAPS=1 pnpm exec playwright test builder-locais-real-maps`. Ele confirma que `.gm-style` monta, que não há `.gm-err-container`/auth error, que os pins reais foram plotados e que clicar num pin (via hook dev-only `window.__catalogMapTest.clickMarker`, eliminado no build de produção por `import.meta.env.DEV`) abre o `InfoWindow` real (`.gm-style-iw`) com o mesmo preço da lista e adiciona o local. **Checklist manual** (alternativa quando não der pra rodar o gated test): abrir `/montar-campanha` autenticado como anunciante → trocar para "Mapa" → confirmar que o mapa renderiza (não há overlay de erro), que aparecem pins, que clicar num pin abre o popup com preço/período corretos e que "Adicionar" coloca o local no plano.
- **Client Management**: Detailed internal view for advertisers (`/clientes/:id`) covering campaigns, quotations, finances, CRM contacts, account hierarchy (parent-child), and service orders. Leads are auto-classified as "upsell" or "new" based on existing client matching.
- **Pricing Engine**: Cost-based pricing with markup, supporting multiple products with volume-based pricing tiers, dynamic unit labels, and product-specific commission/tax rates. Features a multi-product budget creator and simulator.
- **Digital Signature**: Public signing flow for quotations and service orders, updating status, creating campaigns, and generating branded PDF contracts with audit-ready content hashing.
- **Contas a Pagar — Ledger Único (Finrefac #1–#10)**: Toda saída financeira vive em `accounts_payable` com `sourceType` semântico (`restaurant_commission`, `vip_repasse`, `supplier_cost`, `freight_cost`, `partner_commission`, `seller_commission`, `tax`, `manual`), `competenceMonth` (YYYY-MM) e `sourceRef` jsonb. Materializadores idempotentes: `createInvoice` → AP `tax`; `markInvoicePaid` → AP `partner_commission` (agregado por `(partnerId, month)` sob advisory lock) + `vip_repasse` (quando produto é digital); `restaurant_payments` espelhados em AP `restaurant_commission`. Regra "tela vs bolacha": produtos digitais pulam comissão restaurante / produção / frete e geram repasse VIP. Conciliação bancária 1↔1 e 1↔N (`bank.reconcile` / `bank.undoReconcile`) com restauração de `prevStatus`/`prevReceivedDate`/`prevPaymentDate`. Trilha de auditoria via wrapper `audited()` em `audit_log` (entityType: invoice / accounts_payable / restaurant_payment / bank_transaction). Documentação canônica: `docs/financeiro-glossario.md` (definições + mapeamento de colunas) e `docs/financeiro-fluxograma.md` (Mermaid do ciclo completo).
- **Bonificação**: Campaigns and quotations can be marked as `isBonificada`, excluding them from financial KPIs and aggregations.
- **Campaign Batches**: Divides the year into 13 four-week cycles, used for all campaign/quotation/OS period definitions, with event-aware labels.
- **System Version Tag**: Displays build-time version and environment (`dev`/`prod`) in the UI.
- **Banco de testes E2E isolado — defesa em camadas (Task #189 + #202; desacoplado do deploy na #219)**: O `scripts/pre-deploy.sh` sobe um dev server para rodar Playwright; os endpoints `/api/dev-*` fazem INSERT real (parceiros/clientes/campanhas "E2E ..."), então qualquer falha de configuração historicamente vazou lixo pra produção. Desde a Task #219 esse script NÃO faz mais parte do build de deploy (que agora só roda `npm run build`) — ele é executado MANUALMENTE antes de publicar. As quatro camadas independentes abaixo continuam protegendo qualquer execução do script (manual ou futura), cada uma sozinha já barrando o estrago:
  1. **Módulo isolado + flag de opt-in.** As rotas `/api/dev-*` vivem em `server/_dev/devEndpoints.ts` e só são `dynamic import`-adas quando `process.env.DEV_FIXTURES === "1"`. `npm run start` (produção) NUNCA seta essa flag — o esbuild até bundla o módulo, mas ele nunca é alcançado. `pnpm run dev` e o `pre-deploy.sh` setam a flag explicitamente.
  2. **Sentinel no banco de teste.** A migration `create_e2e_test_db_sentinel_task_202` cria a tabela `e2e_test_db_sentinel(allowed boolean primary key)` em TODOS os bancos. A LINHA `(true)` só é inserida pelo `devEndpoints.ts` no boot, e somente se o host da `DATABASE_URL` atual não bate com nenhum padrão de produção conhecido (`getForbiddenPatterns()` → env var `PROD_DB_HOSTS`, CSV de substrings; fallback `.neon.tech`). Antes de qualquer INSERT, cada endpoint faz `SELECT 1 FROM e2e_test_db_sentinel WHERE allowed=true` — sem a linha, responde 503. O banco de produção fisicamente não tem o sentinel.
  3. **Comparação por host:porta:database no pré-deploy.** O `pre-deploy.sh` parseia `DATABASE_URL_TEST` e `DATABASE_URL` com `new URL(...)` e aborta se host:porta:database baterem (querystring irrelevante). Também aborta se o host de teste cair em qualquer padrão proibido (mesma lista que o módulo dev).
  4. **DATABASE_URL_TEST automático via PG*.** O script resolve `DATABASE_URL_TEST` a partir dos secrets `PG*` (Postgres nativo Replit, isolado do Neon prod) quando o secret explícito não está setado, e exporta `DATABASE_URL=$DATABASE_URL_TEST` apenas no subprocess do dev server. O `DATABASE_URL` do build segue apontando para Neon prod para o `npm run build` subsequente.
  5. **Guard no `pnpm run test:e2e` standalone (Task #220).** Quando a suíte é rodada SEM `E2E_BASE_URL` (caminho standalone), o próprio `playwright.config.ts` sobe o dev server — e antes esse server herdava a `DATABASE_URL` do ambiente, que num build/deploy seria produção. O módulo `e2e/test-db.ts` (`resolveTestDatabaseUrl()`) replica as camadas 3+4 em TypeScript: resolve `DATABASE_URL_TEST` (ou deriva de `PG*`), valida que (a) está presente, (b) é uma URL postgres válida, (c) NÃO é, por host:porta:database, o mesmo banco da `DATABASE_URL` atual, (d) o host NÃO bate com `PROD_DB_HOSTS`. Falha lançando erro claro no load do config (antes de qualquer teste). Quando válido, injeta `DATABASE_URL=<banco de teste>` + `DEV_FIXTURES=1` no `webServer` do Playwright, garantindo que o dev server da suíte aponte SEMPRE para o banco isolado. Com `E2E_BASE_URL` setado (ex.: `pre-deploy.sh`), o `webServer` é desligado e a validação fica a cargo do script bash.

  **Discriminador prod vs teste (`PROD_DB_HOSTS`).** Em produção, prod e o banco de teste E2E são DUAS instâncias Neon distintas (mesmo provedor `.neon.tech`), então o fallback genérico `.neon.tech` rejeitaria erroneamente o banco de teste. A env var `PROD_DB_HOSTS` (escopo **shared**, valor = substring única do host de PRODUÇÃO, ex.: `empty-scene-ae775drh`) é a fonte de verdade que distingue os dois — lida tanto pelo `devEndpoints.ts`/`e2e/test-db.ts` quanto pelo `pre-deploy.sh`. Mantê-la em `shared` (não só em `production`) é o que faz o guard e o pré-deploy aceitarem o banco de teste Neon dedicado em dev/build, enquanto continuam barrando o host de produção.

  **Configurando o secret `DATABASE_URL_TEST` localmente.** O banco de teste é uma instância Postgres dedicada, separada da produção. Para apontar a suíte para ela: defina o secret `DATABASE_URL_TEST` (aba Secrets do Replit ou `setEnvVars`) com a connection string completa do banco de teste isolado. Se preferir usar o Postgres nativo do Replit como banco de teste, basta NÃO setar `DATABASE_URL_TEST` — o guard e o `pre-deploy.sh` derivam a URL automaticamente dos secrets `PG*` (`PGHOST/PGUSER/PGPASSWORD/PGDATABASE/PGPORT`). Em qualquer caso, se a configuração resultante apontar para produção (host em `PROD_DB_HOSTS`) ou para o mesmo banco da `DATABASE_URL`, a suíte aborta com mensagem explícita antes de rodar.

  Schema do banco de teste é criado automaticamente por `runMigrations()` na primeira inicialização. Cleanup retroativo do lixo já emitido é feito por `inactivate_leaked_e2e_records_task_189` + `inactivate_leaked_e2e_records_task_202` (inativam em vez de DELETE para preservar trilha financeira de invoices reais associadas aos clientes E2E).
- **URL pública canônica**: O servidor resolve a URL pública via `server/_core/appUrl.ts` (helper `appUrl()`). Prioridade: `APP_URL` env → fallback hardcoded `https://app.mesaads.com.br` em produção → `REPLIT_DEV_DOMAIN` em dev. Use SEMPRE esse helper para construir links externos (convites Clerk, callbacks OAuth, links de assinatura digital) em vez de ler `process.env.APP_URL` direto — caso contrário, links em produção podem cair no workspace dev se o secret não estiver configurado. Em produção, manter `APP_URL=https://app.mesaads.com.br` setado é a configuração recomendada (o fallback é apenas rede de segurança).

### System Design Choices
- **Monorepo Structure**: Separation into `client/` (frontend), `server/` (backend), and `shared/` (common types and constants).
- **tRPC**: Ensures end-to-end type safety between frontend and backend APIs.
- **Drizzle ORM**: Provides type-safe database interactions and schema definition.
