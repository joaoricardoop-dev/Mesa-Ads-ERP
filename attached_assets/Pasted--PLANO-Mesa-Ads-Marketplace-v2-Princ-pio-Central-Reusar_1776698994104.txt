```sql
CREATE TABLE seasonal_multipliers (
  id serial PRIMARY KEY,
  product_id integer REFERENCES products(id),
  restaurant_id integer REFERENCES active_restaurants(id), -- null = global
  season_label varchar(100) NOT NULL,
  start_date date NOT NULL,

```

## 3. FLUXO DETALHADO: /montar-campanha (NOVA LÓGICA)

### Step 1 — Explorar Locais
- **Componente:** novo `StepLocais` (extraindo lógica do `RestaurantsMap.tsx` + `ActiveRestaurants.tsx`)
- **Dados:** query `anunciantePortal.listAvailableLocations(productType?, neighborhood?, startDate?, endDate?)`
  - Retorna: restaurantes com `productLocations` ativos, shares disponíveis calculados, rating, audiência
- **UI:** Toggle mapa/lista. Filtros: bairro, tipo de público, produto disponível. Cards com: nome, bairro, audiência estimada, produtos disponíveis, shares livres.
- **Ação:** selecionar 1..N locais → avançar

### Step 2 — Produtos por Local
- **Componente:** novo `StepProdutosPorLocal`
- **Para cada local selecionado:** mostrar `productLocations` disponíveis com shares restantes
- **Dados:** query `anunciantePortal.getProductsForLocations(restaurantIds[], startDate, endDate)`
  - Cruza `productLocations` com `campaignItems` do período para calcular ocupação real
- **UI:** Accordion por local → dentro de cada um, cards de produto com shares disponíveis e preço por share
- **Ação:** para cada local, escolher produto e quantidade de shares

### Step 3 — Configurar Duração e Ciclos
- **Componente:** `StepConfigurar` (existente, adaptado)
- **Lógica:** semanas → ciclos automáticos de 4 semanas → preço total
- **Multiplicador sazonal:** se período cobre datas com `seasonal_multipliers`, mostrar aviso e preço ajustado
- **Desconto por duração:** `DESCONTOS_PRAZO` existente (4w=0%, 8w=3%, 12w=5%...)

### Step 4 — Revisar Carrinho
- **Componente:** `StepConfirmacao` (existente, adaptado)
- **Persistência:** salvar em `campaign_drafts` ao avançar cada step
- **Agrupamento:** por local → por produto → shares × ciclos × preço

### Step 5 — Submeter
- **Dual-track:**
  - Self-service: chama `quotation.createFromBuilder` (existe) com novos campos (restaurantIds, shareIndex, cycleWeeks)
  - Vendedor interno: fluxo de cotação existente
- **Pós-submit:** status "aprovacao" → notificação para equipe interna → aprovação → campanha criada com `bulkCreate` de fases

## 6. INTEGRAÇÃO GOOGLE DRIVE (documentos financeiros)

### Contexto
O sistema já usa Google Cloud Storage (GCS) via `logoUploadRouter.ts`. O Google Drive é outro produto do mesmo ecossistema — integração natural e sem fricção para a equipe que já gerencia arquivos lá.

### Abordagem: Google Drive Picker API (nível rico)

**Por quê Picker e não só colar link:**
- Equipe não precisa sair do sistema para buscar o arquivo no Drive
- Picker garante que o arquivo é real (não um link quebrado)
- Retorna metadados (nome, tipo, última modificação) além do link
- Integração leve — 1 script JS, sem backend adicional

**Fluxo no admin (equipe Mesa):**
1. Ao editar uma invoice/fatura, há um botão "Vincular documento do Drive"
2. Abre o Google Picker (overlay modal no navegador)
3. Equipe navega nas pastas do Drive e seleciona o arquivo (PDF, NF, recibo)
4. Sistema salva: `documentUrl` = link compartilhável do Drive, `documentLabel` = nome do arquivo
5. Arquivo precisa estar compartilhado como "qualquer pessoa com o link pode ver" (ou usar Service Account)

**O que o anunciante vê:**
- No portal, lista de documentos com nome e botão "Abrir" → abre o arquivo no Google Drive
- Nenhuma cópia é feita — o Drive é a única fonte

**Requisitos técnicos:**
- `GOOGLE_API_KEY` + `GOOGLE_CLIENT_ID` (OAuth) para o Picker — mesmas credenciais do ecossistema GCS se já configuradas
- Script: `https://apis.google.com/js/api.js` carregado no frontend admin (não no portal do anunciante)
- 1 componente `DriveFilePicker.tsx` — botão + callback com metadados do arquivo selecionado
- Sem novo campo de schema além do `documentUrl` já previsto em `invoices`

**Alternativa simples (fallback):**
- Se OAuth/credenciais forem complicadas de configurar, manter campo `documentUrl` como input de texto livre — equipe cola o link do Drive manualmente. Mesmo resultado, zero integração.

**Decisão de implementação:** começar com a alternativa simples (input de texto) e evoluir para o Picker se a equipe sentir fricção.

- **Cancelamento:** política a definir no futuro

- **Renovação automática:** lógica de prioridade a implementar após ciclo básico
