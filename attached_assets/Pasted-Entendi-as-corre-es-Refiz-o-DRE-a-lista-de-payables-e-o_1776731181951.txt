# Prompt para Replit — Refatorar Aba "Financeiro" dentro do Batch (v2)

## Problema

Em [CampaignDetail.tsx](client/src/pages/CampaignDetail.tsx) linhas **3028-3550+**, a aba **Financeiro** do batch hoje:

1. **Mistura escopo** — mostra parâmetros, DRE total (Mensal + Total), faturas e contas a pagar da campanha inteira. Em 6 batches, os mesmos dados se repetem 6 vezes.
2. **Duplica o Consolidado** — que já é a fonte de verdade da campanha.
3. **DRE está errado** — aplica comissão restaurante mesmo em telas, separa "comissão parceiro" (conceito extinto), não aplica gross-up no BV, e ordena as deduções fora do padrão contábil.
4. **Economics não são editáveis por batch** — um batch com frete especial ou desconto pontual não tem onde registrar.

## Objetivo

Aba Financeiro do batch passa a ter exatamente 4 blocos, **no escopo exclusivo do batch**:

1. Economics do Batch (editáveis com override sobre a campanha)
2. DRE do Batch (estrutura contábil correta, condicional por tipo de produto)
3. Custos e Repasses do Batch (auto-gerados quando emite fatura do batch)
4. Fatura do Batch (única, 1 batch = 1 mês = 1 fatura)

## DRE do Batch — estrutura correta

**Ordem fixa das deduções** (a aba atual está fora de ordem):

```
Receita Bruta (faturamento do batch)
(-) Impostos                                    [alíquota × receita]
= Receita após Impostos
(-) Comissão do Canal                           [exclusiva por tipo de produto ⬇]
= Base
(-) Custo Frete
(-) Custo Produção
= Base de Contribuição
(-) BV da Campanha                              [com gross-up, só se há agência/parceiro]
= Lucro Líquido
Margem (%)
```

### Regra condicional da "Comissão do Canal" (mutuamente exclusiva)

| Tipo do produto do batch | Linha exibida | Origem do % |
|---|---|---|
| `coaster` / `bolacha` / físico | **Comissão Restaurante** (% × Receita após Impostos) | `restaurantCommission` da campanha, override do batch, ou % do próprio restaurante |
| `telas` / `janelas_digitais` | **Repasse Sala VIP** (% × Receita após Impostos) | `vipProviderCommissionPercent` do produto, override do batch |
| outro / bonificada | linha não é exibida | — |

- Nunca os dois ao mesmo tempo. A UI deve exibir apenas a linha aplicável — a outra nem renderiza.
- Se for bolacha e houver múltiplos restaurantes no batch, a comissão é **rateada por restaurante** (1 linha de payable por restaurante), mas no DRE aparece consolidada numa única linha (soma).

### BV da Campanha (substitui "Comissão Parceiro")

- A linha antiga **"Comissão Parceiro"** deixa de existir — renomear para **"BV da Campanha"** em schema, UI, payables e glossário.
- Aplica **gross-up**: o valor exibido e efetivamente pago é inflado para compensar tributos sobre o BV.
- Fórmula: `bvBruto = baseContribuicao × (bvPercent / 100) / (1 - grossUpRate)`
  - `grossUpRate` default = taxa efetiva configurada em parâmetro do sistema (ex: 14,17% para IRRF+PIS/COFINS quando aplicável) — deixar como config global em `finance.grossUp.bvTaxRate`, com override por campanha/parceiro
- No DRE, exibir duas sub-linhas legíveis sob "BV da Campanha":
  ```
  (-) BV da Campanha (gross-up)               R$ X.XXX
      ├─ BV líquido (N%)                       R$ Y.YYY
      └─ Gross-up tributário                   R$ Z.ZZZ
  ```
- Só aparece se `campaign.partnerId` existe e `bvPercent > 0`.

### Campanha Bonificada

- Sem receita → DRE simplificado: só mostra os custos (Produção + Frete + Comissão canal se houver) somados na linha "Total Bonificação" negativo.

## Lógica de cálculo — invariantes obrigatórias

Criar/atualizar `calcPhaseFinancials` em [server/finance/calc.ts](server/finance/calc.ts):

```ts
function calcPhaseFinancials(phaseId): {
  receitaBruta,
  impostos,
  receitaAposImpostos,
  canalTipo: 'restaurante' | 'sala_vip' | null,
  canalValor,                       // comissão rest OU repasse VIP (nunca os dois)
  canalPercent,
  base,                             // receitaAposImpostos - canalValor
  custoProducao, custoFrete,
  baseContribuicao,
  bvAplicavel: boolean,
  bvPercent, grossUpRate,
  bvLiquido, bvGrossUp, bvTotal,    // breakdown
  lucroLiquido, margemPct,
  effectiveParams                   // quais vieram de override × quais herdados
}
```

**Invariantes testadas:**
- Produto bolacha → `canalTipo === 'restaurante'`, nunca `'sala_vip'`
- Produto tela/janela → `canalTipo === 'sala_vip'`, nunca `'restaurante'`
- `SUM(calcPhaseFinancials(p).lucroLiquido for p in campaign.phases) === calcCampaignConsolidated(campaign).lucroLiquido` (até o centavo)
- `bvTotal === bvLiquido + bvGrossUp` (idempotência do gross-up)
- Sem BV configurado → `bvAplicavel === false` e linha some do DRE
