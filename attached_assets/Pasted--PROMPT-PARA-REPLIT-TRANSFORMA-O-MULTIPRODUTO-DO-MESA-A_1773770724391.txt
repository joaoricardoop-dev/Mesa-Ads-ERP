 PROMPT PARA REPLIT — TRANSFORMAÇÃO MULTIPRODUTO DO MESA ADS ERP
 
## CONTEXTO DO PROJETO
 
Este é um ERP para a Mesa Ads, uma empresa de mídia indoor que **atualmente opera apenas com um produto: Bolachas de Chopp (coasters)**. Todo o sistema — banco de dados, precificação, cotações, campanhas, ordens de serviço e financeiro — foi construído em torno desse único produto.
 
**A missão agora é transformar a plataforma em um sistema multiproduto**, onde coasters são apenas um dos vários produtos disponíveis (ex: displays de mesa, cardápios patrocinados, totens digitais, adesivos, porta-guardanapos, etc.).
 
---
 
## STACK ATUAL
 
- **Frontend**: React + TypeScript + Vite + Tailwind + shadcn/ui
- **Backend**: Node.js + Express + tRPC
- **Banco de Dados**: PostgreSQL + Drizzle ORM
- **Estrutura**: client/ + server/ + shared/ + drizzle/

## DIAGNÓSTICO: ONDE O SISTEMA ESTÁ ACOPLADO A "COASTERS"
 
### 1. SCHEMA DO BANCO (drizzle/schema.ts)
 
Os seguintes campos estão hardcoded para coasters e precisam ser generalizados:
 
| Tabela | Campos acoplados | Problema |
|--------|-----------------|----------|
| `campaigns` | `coastersPerRestaurant`, `usagePerDay`, `daysPerMonth` | Assume que todo produto é coaster distribuído por restaurante |
| `campaignRestaurants` | `coastersCount`, `usagePerDay` | Alocação fixada em coasters |
| `quotations` | `coasterVolume` | Volume atrelado a coasters |
| `quotationRestaurants` | `coasterQuantity` | Quantidade por restaurante fixada em coasters |
| `activeRestaurants` | `coastersAllocated` | Inventário por restaurante assume coasters |
| `serviceOrders` | `coasterVolume`, `networkAllocation` | Produção atrelada a coasters |
| `restaurants` | `coastersAllocated` | Alocação fixa em coasters |
| `products` | `defaultQtyPerLocation` | Funciona, mas assume distribuição por local |
| `productPricingTiers` | `custoUnitario`, `frete`, `margem` | Funciona, mas só existe pricing para coasters |
 
### 2. LÓGICA DE PRECIFICAÇÃO (server/db.ts + server/quotationRouter.ts)
 
A função `calcPricing` e toda a lógica de custo usa:
```
unitCost = (custoUnitario × artes) + (frete / totalVolume)
custoPD = productionCost + fixedCommission
custoBruto = custoPD / (1 - totalVarRate)
```
Essa lógica funciona para qualquer produto, MAS está chamada apenas no contexto de coasters. Precisa ser parametrizada por `productId`.
 
### 3. SIMULADOR DE PREÇO (client/src/hooks/useSimulator.ts — 800+ linhas)
 
O simulador é o coração da precificação e está 100% acoplado a coasters:
- Input: `coastersPerRestaurant`, `usagePerDay`, `daysPerMonth`
- Impressões = `coastersPerRestaurant × usagePerDay × daysPerMonth`
- Tabela de markup com 9 cenários fixos para coasters
- Tabela de desconto por volume de coasters
- Gráficos de receita baseados em coasters/restaurante
 
### 4. INTERFACE (client/src/pages/ e components/)
 
| Arquivo | Linhas | Problema |
|---------|--------|----------|
| `PriceTable.tsx` | ~1106 | Toda a UI do simulador referencia coasters |
| `Quotations.tsx` | ~1315 | Lista cotações por `coasterVolume` |
| `QuotationDetail.tsx` | ~1024 | Edição referencia volume de coasters |
| `Campaigns.tsx` | Muito grande | CRUD de campanhas assume coasters |
| `CampaignDetail.tsx` | Muito grande | Alocação de coasters por restaurante |
| `ServiceOrders.tsx` | ~705 | Ordens de produção para coasters |
| `Products.tsx` | ~497 | Gerenciamento de produto (mais genérico) |
| `RestaurantAllocationPanel.tsx` | ~550 | Distribui coasters por restaurante |
| `InputPanel.tsx` | — | Controles do simulador (sliders de coasters) |

## RESULTADO ESPERADO
 
Ao final, o sistema deve permitir:
 
1. **Cadastrar múltiplos produtos** (coasters, displays, cardápios, totens, etc.) cada um com sua própria tabela de preços
2. **Criar orçamentos com múltiplos produtos** — um único orçamento para um cliente pode ter 3.000 bolachas de chopp + 50 displays de mesa + 20 cardápios
3. **Cada item do orçamento calcula seu preço** individualmente com base nos tiers do respectivo produto
4. **Campanhas gerenciam múltiplos produtos** com alocação por local quando aplicável
5. **O simulador de preço fica em Configurações** como ferramenta de planejamento, podendo simular qualquer produto
6. **Ordens de serviço detalham itens** por produto com status individual
7. **Financeiro discrimina** receita e custo por tipo de produto
